Electron IPC Security Hardening

ClaudeDeck · fixes the Fable adversarial audit · 2 CRITICAL + 3 HIGH + 1 MEDIUM · scope = electron/ only

A · Read this first

Goal. Close every renderer-reachable path to code execution, permission-gate bypass, SSRF, and arbitrary file write — without breaking the defenses that already hold or the shipped bypassPermissions UX. The renderer is treated as untrusted (realistic threat: XSS from a model-rendered message), so policy moves into the main process and the spawn transport stops handing attacker tokens to cmd.exe.

Key reframing vs the raw audit (a decision, not a gap).
The audit's CRITICAL-2a fix ("never accept bypassPermissions from the renderer") is wrong for this app: bypassPermissions is a legitimate user-selected mode (permissionModes.ts:17, ModePicker + voice command "บายพาส"). Banning it breaks a real feature. The actual threat is a compromised renderer. So we split CRITICAL-2:
  • 2a → add a strict CSP (main-process header injection) so model output can't run script in the renderer in the first place. We do not restrict the mode the user picks.
  • 2b → stop trusting renderer-supplied updatedInput on allow: the main process remembers the exact tool input it parsed from the CLI's can_use_tool request and echoes that back, so a forged "allow" can't silently rewrite a tool call. This is invisible to honest UX.

T1 · electron/spawnCli.ts — safe Windows CLI spawn CRITICAL-1

One choke point for spawning a resolved CLI binary so attacker-controlled argv never reaches a shell parser. For a real .exe, spawn directly (shell:false) — CreateProcess + the child's own CommandLineToArgvW, zero shell layer. For an npm .cmd/.bat shim (what where claude usually returns), the only way to launch is via cmd.exe, so we use windowsVerbatimArguments:true and quote each arg ourselves with quoteForCmd: wrap in ", escape inner quotes per CommandLineToArgvW backslash rules, and neutralize % (the one metacharacter cmd still expands inside quotes).

  • [ ] RED spawnCli.test.ts: quoteForCmd('a&calc') → "a&calc" (wrapped, & inert); quoteForCmd('x"y') escapes the quote; quoteForCmd('%PATH%') throws (callers must pre-reject %); isDirectExe('C:/x/claude.exe') → true; isDirectExe('C:/x/claude.cmd') → false.
  • [ ] GREEN implement:
import { spawn, type ChildProcess, type SpawnOptions } from 'node:child_process'

const CMD_SHIM = /\.(cmd|bat)$/i

/** True when the resolved bin is directly executable by CreateProcess (no shell). */
export function isDirectExe(bin: string): boolean {
  return process.platform !== 'win32' || !CMD_SHIM.test(bin)
}

/** Quote one argv token for `cmd.exe` with windowsVerbatimArguments.
 *  Inside double quotes cmd suppresses & | < > ^ ( ). `%` (var-expansion) CANNOT
 *  be reliably escaped inside quotes under verbatim args, so callers MUST reject
 *  `%` upstream (T3) — quoteForCmd asserts it never receives one. */
export function quoteForCmd(arg: string): string {
  if (arg.includes('%')) throw new Error('refusing to quote a token containing %')
  // Backslashes that precede a quote (or the closing quote) must be doubled
  // per CommandLineToArgvW so the child re-parses the original token exactly.
  const s = arg.replace(/(\\*)"/g, '$1$1\\"').replace(/(\\*)$/, '$1$1')
  return `"${s}"`
}

/** Spawn a resolved CLI binary safely on every platform. */
export function spawnCli(bin: string, args: string[], opts: SpawnOptions = {}): ChildProcess {
  if (isDirectExe(bin)) return spawn(bin, args, { windowsHide: true, ...opts })
  const line = [bin, ...args].map(quoteForCmd).join(' ')
  return spawn('cmd.exe', ['/d', '/s', '/c', line], {
    windowsHide: true, windowsVerbatimArguments: true, ...opts,
  })
}
Windows reality (from review): where claude almost always returns claude.cmd (npm shim), so on Windows the .cmd branch is taken every time — quoteForCmd is the sole defense, which is why the % reject in T3 is mandatory, not optional. No legitimate token needs %: the --settings JSON has none, tool rules like Bash(git *) have none, dirs are paths. Defense-in-depth: probe() may prefer a .exe result over .cmd when where lists both.

  • [ ] RED netGuard.test.ts: https://api.x.com ok; http://x rejected (scheme); https://127.0.0.1, https://169.254.169.254, https://10.0.0.5, https://[::1], https://localhost all blocked; bare hostname with private literal blocked.
  • [ ] GREEN:
const PRIVATE_V4 = [/^127\./,/^10\./,/^192\.168\./,/^169\.254\./,/^172\.(1[6-9]|2\d|3[01])\./,/^0\./]
function isBlockedHost(host: string): boolean {
  const h = host.replace(/^\[|\]$/g, '').toLowerCase()
  if (h === 'localhost' || h === '::1' || h.startsWith('fe80:') || h.startsWith('fc') || h.startsWith('fd')) return true
  if (/^\d{1,3}(\.\d{1,3}){3}$/.test(h)) return PRIVATE_V4.some((re) => re.test(h))
  return false
}
/** https only, public host only. Returns null when safe, else a reason string. */
export function rejectUnsafeUrl(raw: string): string | null {
  let u: URL
  try { u = new URL(raw) } catch { return 'invalid url' }
  if (u.protocol !== 'https:') return 'only https allowed'
  if (isBlockedHost(u.hostname)) return 'host not allowed'
  return null
}
DNS-rebinding (public name → private A record) is out of scope for this pass; note it as a follow-up. The guard blocks literal-IP SSRF, which is the audit's PoC.

// module-level, keyed by turnId → (request id → original input):
const pendingInput = new Map<string, Map<string, unknown>>()
// on 'permission' in apply(): (pendingInput.get(turnId) ?? pendingInput.set(turnId, new Map()).get(turnId))!.set(req.id, req.input)
// on proc 'exit' (next to turns.delete): pendingInput.delete(a.turnId)   // <-- prevent map leak on crash/cancel
export function respondPermission(turnId, id, decision, opts) {
  const proc = turns.get(turnId); if (!proc?.stdin || proc.stdin.writableEnded) return false
  const original = pendingInput.get(turnId)?.get(id)
  // Echo the input MAIN parsed, not the renderer's. Fall back to opts.input only on a
  // map miss (race) so honest behaviour is preserved; a compromised renderer can't
  // substitute when the original is present.
  const safeOpts = decision === 'allow' ? { input: original ?? opts?.input ?? {} } : { message: opts?.message }
  proc.stdin.write(buildControlResponse(id, decision, safeOpts) + '\n', 'utf8')
  pendingInput.get(turnId)?.delete(id)
  return true
}
Confirmed by review (App.tsx:707): the renderer sends back req.input unchanged on allow — there is no "edit tool input before allow" UX. So echoing the main-stored original breaks nothing today, and the opts?.input fallback covers the map-miss race.
